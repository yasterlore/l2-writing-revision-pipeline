# Frozen Policy Generation Artifact Body Generation Design

This document designs a future artifact body generation boundary for the
frozen policy generation artifact writer. It is docs-only. It does not
implement artifact body generation, generate an artifact body, write artifact
files, implement a manifest writer, compute metrics, evaluate performance, or
claim real-data or production readiness.

## 1. Purpose

The purpose of this document is to define what a future artifact body may
contain, what it must never contain, how it should be audited, how it should
fail closed, and how it should relate to the existing metadata-only artifact
writer, manifest work, runtime smoke checks, and release-quality checks.

This is not:

- artifact body generation implementation
- generated policy body generation
- manifest body generation
- artifact file writing
- manifest file writing
- manifest writer implementation
- output file option implementation
- performance evaluation
- real-data readiness
- production readiness

## 2. Current State

- The artifact writer metadata-only skeleton exists.
- The artifact writer CLI exists.
- The artifact writer runtime Makefile target exists.
- The artifact writer runtime smoke is included in release-quality.
- The artifact writer fixture validator target is included in release-quality.
- Artifact body generation does not exist.
- Manifest body generation does not exist.
- Artifact file writing does not exist.
- Manifest file writing does not exist.

## 3. Artifact Body Definition

An artifact body is a future human-readable or machine-readable summary
generated by the artifact writer. The first allowed body shape should be a
synthetic-only metadata summary.

The artifact body must not include:

- raw learner text
- raw rows
- logits or probabilities
- private paths
- performance metrics
- generated policy body content
- request, pointer, or expected result bodies

The body is not the manifest. It is not the generated frozen policy. It is not
a learner record. It is not a performance report.

## 4. Allowed Content

Allowed content candidates for a future safe metadata body:

- schema version
- artifact_id
- manifest_id
- writer_version
- synthetic notice
- no-oracle notice
- validation reference IDs
- artifact flags
- safety flags
- count summary
- reason codes
- failed checks
- safe metadata field names
- safe IDs
- created_by_version
- body_suppression policy summary
- file_writing policy summary

Allowed fields must remain metadata-level. They may describe counts, flags,
safe IDs, schema versions, and safety policy outcomes, but not payload bodies
or learner content.

## 5. Forbidden Content

The future artifact body must fail closed if it contains or attempts to carry:

- raw learner text
- raw event rows
- revision event raw rows
- micro episode raw rows
- observed_after_text
- final_text
- gold label
- expected action payload
- scoring feedback payload
- generated policy body
- frozen policy body
- request body
- pointer body
- expected result body
- artifact writer request JSON body
- generator result pointer JSON body
- expected artifact writer result JSON body
- manifest body
- logits
- probabilities
- model scores
- performance metric body
- private paths
- absolute local paths
- real participant metadata
- GitHub raw logs
- full job output

Forbidden content is forbidden in stdout, stderr, docs, generated body fields,
manifest references, validation summaries, and future status markers.

## 6. Artifact Body Schema Design

Proposed future top-level metadata fields:

- artifact_body_schema_version
- artifact_id
- manifest_id
- writer_version
- body_type
- body_status
- content_suppressed
- synthetic_only_notice
- no_oracle_notice
- allowed_sections
- safety_summary
- count_summary
- validation_references
- non_proof_notice

Proposed `body_status` values:

- `suppressed_metadata_only`
- `generated_safe_metadata_body`
- `fail_closed`

Initial implementation policy:

- Artifact body generation remains unimplemented for now.
- When body generation is implemented later, the first generated body mode
  should be `generated_safe_metadata_body` only.
- Raw text body generation should not be implemented.
- The default writer CLI and Makefile runtime smoke should continue to
  suppress body output unless a later safe option is explicitly designed.

## 7. Safety Audit Design

A future artifact body generator should audit the generated safe metadata body
before returning it. The audit should check both structured fields and known
forbidden marker fragments.

Required audit outcomes:

- body_field_count is explicit
- raw_row_count=0
- logits_dump_count=0
- private_path_count=0
- performance_metric_count=0
- request_body_count=0
- pointer_body_count=0
- expected_body_count=0
- artifact_body_allowed=true only for safe metadata body
- body_suppression_checked=true
- artifact_policy_checked=true

The audit should be deterministic and should fail closed before any CLI or
Makefile target can expose unsafe output.

## 8. Fail-Closed Reason Codes

Future fail-closed reason codes should include:

- raw_learner_text_in_artifact_body
- raw_rows_in_artifact_body
- logits_dump_in_artifact_body
- private_path_in_artifact_body
- performance_claim_in_artifact_body
- request_body_leakage
- pointer_body_leakage
- expected_result_body_leakage
- generated_policy_body_leakage
- manifest_body_leakage
- unsafe_artifact_body_schema
- missing_synthetic_notice
- missing_no_oracle_notice
- unknown_artifact_body_schema_version

These codes describe body safety failures. They are not model performance
failures, production readiness signals, or artifact educational quality
scores.

## 9. Relation To Current Artifact Writer

The current artifact writer returns a metadata-only result. It suppresses
artifact body output and does not write files.

Future artifact body generation should be optional. The default should remain
body-suppressed until a later safe design and implementation explicitly add a
safe metadata body mode.

The current writer boundaries should remain:

- CLI does not output artifact body by default
- Makefile runtime target does not output artifact body by default
- release-quality continues to check body suppression
- fixture validation continues to check metadata-only contracts

## 10. Relation To Manifest Writer

Artifact body generation and manifest body generation are separate concerns.

Future manifest work may reference artifact body metadata, but it should not
embed raw artifact body content. The manifest should record safe body
metadata only, such as body schema version, body status, suppression status,
safe IDs, and audit flags.

Manifest body generation remains a separate future step.

## 11. Relation To Release-Quality

Artifact body generation should not be added to release-quality until these
exist:

- standalone body fixtures
- a body validator
- CLI behavior with no-body-leakage tests
- a standalone Makefile target
- log safety review

The first release-quality integration should check safe metadata body
generation only. Success would not mean artifact educational quality,
generated policy quality, model performance, real-data readiness, or
production readiness.

## 13. Future Tests

Future tests should verify:

- safe body is JSON serializable
- output is deterministic
- no raw rows
- no logits
- no private paths
- no raw learner text
- no request, pointer, or expected body
- no manifest body
- invalid cases fail closed
- default CLI does not print body
- optional future body mode remains safe
- Makefile target does not write files by default

Tests must remain synthetic-only and should assert safety using safe labels,
counts, schema versions, and reason codes.

## 16. Beginner-Friendly Explanation

An artifact body is the future visible summary that the artifact writer may
eventually produce. Today the writer only returns metadata. This design
decides what a future body may safely say before any code starts producing
one.

The body must not include raw learner text because that would turn a safe
metadata artifact into a content-bearing artifact. It must also avoid raw
rows, logits, probabilities, private paths, and performance metrics because
those belong to different safety and evaluation boundaries.

File writing is separate because producing a safe in-memory metadata body and
writing files to disk have different risks. Release-quality integration is
also separate because a new body mode needs its own fixtures, validator, CLI
tests, Makefile target, and log safety review first.
