# Summary Manifest Schema Hardening Design

This document designs future schema hardening for the no-config synthetic E2E
summary manifest.

It records schema-hardening design and the initial manifest schema-version
implementation. It does not change tests, scorer logic, scorer weights,
scoring formula, deterministic tie-break behavior, or fixtures.

This is not performance evaluation.

## 1. Purpose

The purpose of this design is to define how `summary.manifest.json` could be
hardened in future steps while staying safe, count-only, and no-config only.

The design focuses on:

- reducing stale-marker and run-mismatch risk
- clarifying which manifest fields should become stricter
- avoiding raw generated bodies in docs, stdout, and marker output
- preserving no-oracle and privacy boundaries
- avoiding performance evaluation or tuning signals

The manifest should remain reliability metadata, not a report body.

## 1.1 Implementation Status

Step 116 implemented the initial manifest schema-version hardening:

- `scripts/run_synthetic_e2e_summary.sh` writes
  `manifest_schema_version="1.0"` into `summary.manifest.json`
- `scripts/check_synthetic_diagnostic_distribution.sh` requires that exact
  version before running distribution checks
- missing, non-string, or unexpected manifest schema versions fail closed as a
  precondition error
- `summary_schema_version` remains separate and describes the summary CSV
  schema compatibility

Step 118 implemented strict allowed-key validation for
`manifest_schema_version="1.0"` in the diagnostic distribution checker:

- unknown manifest keys fail closed as precondition errors
- explicit forbidden body-like keys remain a separate fail-closed reason
- the manifest body is not printed

Step 120 moved the current manifest schema constants into
`scripts/lib/summary_manifest_schema.sh` so the generator and checker can share
the same version and key lists.

Summary hash, per-case diagnostic consistency hardening, and wrapper scripts
are still future work.

For the allowed-key validation design, see
[Summary manifest allowed-key validation design](summary_manifest_allowed_key_validation_design.md).
For the shared schema constants design, see
[Summary manifest schema centralization design](summary_manifest_schema_centralization_design.md).
For the future sync check design, see
[Summary manifest schema sync check design](summary_manifest_schema_sync_check_design.md).

## 2. Current Manifest State

The no-config summary manifest is generated by
`scripts/run_synthetic_e2e_summary.sh` after successful summary generation.

It is required by `scripts/check_synthetic_diagnostic_distribution.sh`.

Current fields:

- `manifest_schema_version`
- `run_id`
- `completed_at`
- `summary_path`
- `case_count`
- `diagnostic_summary_count`
- `content_suppressed`
- `no_config_summary`
- `generator_script`
- `summary_schema_version`

The manifest does not include raw summary bodies, diagnostic bodies, JSONL
rows, candidate score rows, config bodies, or performance metrics.

## 3. Current Validation State

The diagnostic distribution check currently validates:

- manifest file exists
- manifest is valid JSON
- `manifest_schema_version` is the expected value
- all keys are allowed for `manifest_schema_version="1.0"`
- `content_suppressed` is `true`
- `no_config_summary` is `true`
- `case_count` is an integer greater than 0
- `diagnostic_summary_count` is an integer greater than or equal to 0
- `summary_path` matches the expected no-config summary path
- `generator_script` is the expected summary generator script
- forbidden body-like keys are absent
- manifest `case_count` equals summary data-row count

This is required precondition validation, not performance evaluation.

## 4. Additional Schema Fields To Consider

### `manifest_schema_version`

Implemented.

This would version the manifest itself separately from the summary CSV schema.
It makes future validator changes easier to reason about.

### `run_id` Format

Recommended to harden.

Future validation can require a simple safe pattern such as a fixed prefix plus
UTC timestamp and process identifier. It should not include private paths,
usernames, hostnames, or raw input names.

### `started_at`

Optional.

This can help reason about run duration and stale output, but it should not be
used as a performance metric. If added, it should be UTC timestamp metadata
only.

### `completed_at`

Recommended to keep and validate.

Future validation can check UTC format. It should not be used to claim
performance or runtime quality.

### `generated_case_names_count`

Recommended as count-only metadata.

This can clarify the difference between discovered input cases and completed
summary rows without listing raw case names.

### `diagnostic_summary_count`

Recommended to keep.

Future validation can compare it with count-only diagnostic summary statuses
from the summary CSV.

### `summary_sha256`

Optional and cautious.

A hash can provide safe integrity metadata if treated strictly as a checksum.
It should not be printed by default or interpreted as content.

### `case_name_hashes`

Not recommended initially.

Even hashed case names may create unnecessary complexity and may be mistaken
for de-identified content evidence. Count-only fields are preferable for the
initial hardening path.

### `output_root`

Optional.

If added, it should be a stable project-controlled relative path such as
`tmp/synthetic_e2e_summary`, not a user home path or private path.

### `distribution_check_compatible_version`

Recommended later.

This can let the checker fail closed when the manifest was produced by an
incompatible schema. It should stay a simple version label.

## 5. Fields Not To Add

Do not add:

- raw case names if they become risky or unnecessary
- raw summary rows
- diagnostic summary bodies
- JSONL rows
- candidate score rows
- expected action details
- `final_text`
- `observed_after_text`
- `gold_label`
- config body
- performance metrics
- participant identifiers
- private paths
- manual output paths
- real-data paths

The manifest should not become a condensed copy of generated outputs.

## 6. Stale Marker Detection Options

### Option A: Summary mtime Compared With `completed_at`

This is simple but fragile. Filesystem timestamp precision and clock behavior
can vary.

Not recommended as the primary mechanism.

### Option B: Summary Hash Stored In Manifest

This can detect whether the current `summary.csv` matches the manifest.

Recommended as an optional future hardening step, provided the hash is treated
as integrity metadata and not printed by default.

### Option C: `run_id` Appears In Both Summary And Marker

This is stronger but would require changing the summary CSV schema or adding
metadata rows.

Not recommended initially because the current summary CSV should remain case
rows only.

### Option D: Marker Written After Summary Rename Only

This is already the current recommended base behavior.

Recommended as the primary initial stale-risk reduction: the marker is created
only after the final summary file is in place.

### Option E: Wrapper Sequencing

This reduces manual and CI ordering mistakes but does not prove file-level
freshness by itself.

Recommended later as operational hardening, not as the only stale-marker
mechanism.

Initial recommendation: keep Option D, then consider Option B as a cautious
future enhancement. Avoid changing the summary CSV schema solely for run-id
embedding.

## 7. Summary Hash Handling

A summary hash may be safe if handled carefully:

- do not print the summary body
- treat the hash as integrity metadata only
- do not treat the hash as performance or content interpretation
- do not require docs or stdout to show the hash by default
- use it only to compare the manifest with the final summary file
- do not use hashing as an excuse to expose raw generated bodies

Hash verification is future hardening, not current behavior.

## 8. Per-Case Diagnostic Consistency

Future validation can add count-only consistency checks:

- compare `diagnostic_summary_count` with rows where
  `diagnostic_summary_status=ok`
- ensure expected diagnostic summary path count matches summary rows, if
  feasible
- fail closed when an ok row points to a missing diagnostic summary path
- keep output limited to counts and reason labels
- do not print diagnostic summary JSON bodies

The checker may need to read diagnostic summary files for validation, but it
must not print their bodies. A safer first step is to validate counts and file
existence only.

Future tests can cover:

- diagnostic count mismatch
- missing diagnostic summary file for an ok row
- unexpected config-enabled path
- safe stdout only

## 9. Future Validation Roadmap

Phase 1 current:

- required marker basic validation
- manifest schema version validation
- case-count consistency
- forbidden key rejection

Phase 2:

- stricter `run_id` and timestamp format checks
- stricter allowed-key validation

Phase 3:

- optional safe `summary_sha256`
- hash verification against final `summary.csv`
- run-id consistency where possible without changing summary rows

Phase 4:

- per-case diagnostic count consistency
- diagnostic summary path existence checks
- no raw diagnostic body output

Phase 5:

- wrapper or CI command ordering
- manifest-aware release smoke checks

## 10. Output Safety, Privacy, And No-Oracle Policy

Manifest schema hardening must preserve:

- no raw summary body
- no marker body output
- no JSONL body
- no diagnostic body
- no candidate score rows
- no expected action feedback
- no config body
- no real participant data
- no private/manual/real data
- no performance metric
- no tuning signal

The manifest is not training data, not performance evidence, and not scoring
feedback.

## 11. Beginner Explanation

### What Is Schema Hardening?

Schema hardening means making the rules for a file stricter and clearer.

Here, it means deciding which fields `summary.manifest.json` should contain and
how the checker should validate them.

### Why Does The Manifest Need Rules?

Without clear rules, a marker could become stale, incomplete, or accidentally
include unsafe fields.

Rules let the checker fail safely before trusting the summary.

### What Is A Stale Marker?

A stale marker is a marker from an older run that no longer matches the current
summary file.

It can make a check look current when it is actually reading old output.

### What Is A Hash For?

A hash can help confirm that a file matches the manifest without printing the
file body.

It is an integrity check, not a way to understand the content.

### Why Be Careful With Hashes?

Hashes are safer than raw bodies, but they are still metadata about generated
content. They should be used only when they make validation meaningfully safer.
