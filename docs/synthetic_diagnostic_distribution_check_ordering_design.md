# Synthetic Diagnostic Distribution Check Ordering Design

This document describes the execution-order precondition for
`scripts/check_synthetic_diagnostic_distribution.sh`.

It records the ordering design and the minimal precondition check added after
that design. It does not change scorer weights, scoring formula, deterministic
tie-break behavior, E2E pipeline behavior, or fixtures.

This is not performance evaluation.

## 1. Purpose

The purpose of this design is to make the ordering requirement for the
synthetic diagnostic distribution check explicit.

The design should:

- clarify that the distribution check depends on the no-config synthetic E2E
  summary output
- explain what `no_cases` means in different states
- avoid confusion when summary generation and distribution checking are run in
  parallel
- preserve output safety and no-oracle boundaries
- avoid treating diagnostic counts as performance metrics

The check is a count-only wiring and regression check, not a model-quality
evaluation.

## 1.1 Implementation Status

The current script performs a minimal fail-closed precondition check before the
distribution scan:

- missing no-config summary path fails with a safe message
- empty summary file fails with a safe message
- missing or malformed header fails with a safe message
- zero data rows fails as `reason=no_cases`
- config-enabled summary paths are rejected for this no-config check
- raw summary bodies, diagnostic JSON bodies, JSONL bodies, and score rows are
  not printed

The script still expects the no-config summary to be generated first by
`scripts/run_synthetic_e2e_summary.sh`.
That summary generator now writes through `summary.csv.tmp` and renames the
completed temp file to `summary.csv`; completion markers are not implemented
yet.

## 2. Problem Summary

`scripts/check_synthetic_diagnostic_distribution.sh` reads the no-config
summary generated by `scripts/run_synthetic_e2e_summary.sh`.

If the distribution check runs before summary generation finishes, it can see a
missing, empty, or partially written summary and report `no_cases`.

That situation is an ordering or precondition issue. It is not necessarily a
diagnostic logic failure.

When `scripts/run_synthetic_e2e_summary.sh` completes first and
`scripts/check_synthetic_diagnostic_distribution.sh` runs afterward, the
distribution check is expected to pass if the no-config synthetic cases and
diagnostic summaries are healthy.

This can confuse CI or manual runs if the commands are parallelized.

## 3. Expected Execution Order

Run the checks in this order:

1. `scripts/run_synthetic_e2e_summary.sh`
2. `scripts/check_synthetic_diagnostic_distribution.sh`

The first command generates the no-config synthetic summary. The second command
validates count-only diagnostic distribution fields from that summary.

The config-enabled summary collector is separate and should not be treated as
the input for this check.

Expected target:

- `tmp/synthetic_e2e_summary/summary.csv`

Separate config-enabled target:

- `tmp/synthetic_e2e_config_summary/<safe_config_name>/summary.csv`

The diagnostic distribution check discussed here is for the no-config summary.

## 4. Handling `no_cases`

`no_cases` should not be a silent pass.

Interpret it by context:

- If the no-config summary has not been generated, `no_cases` is an expected
  precondition failure.
- If summary generation is still running, `no_cases` may indicate a concurrent
  partial-output read.
- If summary generation completed successfully and `no_cases` still appears,
  investigate it as a real failure.
- If `no_cases` appears once during parallel execution, rerun after summary
  generation completes.
- If `no_cases` repeats after a completed summary run, treat it as a failure.

Do not print or paste the raw summary body while investigating. Use safe
count-only status, path, and row-count information.

## 5. Future Hardening Direction

Future script hardening may include:

- additional precondition coverage in
  `scripts/check_synthetic_diagnostic_distribution.sh`
- more detailed safe row-count and case-count checks
- clearer error wording for missing, empty, or partial summary files
- a partial-output marker or completion marker
- completion marker support in the summary generator
- a wrapper script that guarantees sequential ordering
- documentation updates for CI ordering

This document does not implement completion markers, wrapper scripts, or
additional summary generator changes.

Any future hardening should keep output count-only and avoid raw report bodies.
For the atomic-write and completion-marker design, see
[Synthetic E2E summary atomic write design](synthetic_e2e_summary_atomic_write_design.md).

## 6. CI And Manual Run Guidance

Recommended manual and CI order:

```bash
scripts/run_synthetic_e2e_summary.sh
scripts/check_synthetic_diagnostic_distribution.sh
```

Do not run those two commands in parallel.

If `no_cases` appears:

1. Confirm whether summary generation was still running.
2. Re-run the distribution check after summary generation completes.
3. If it passes, record the first result as an ordering/precondition issue.
4. If it fails again, investigate the summary generation or diagnostic output
   path.

Logs should remain safe and count-only.

## 7. Output Safety Policy

The ordering check and any future hardening must not print or paste:

- raw summary CSV body
- diagnostic summary JSON body
- JSONL body
- candidate score rows
- raw text
- local context
- expected action details
- config JSON body
- real participant identifiers

Allowed output should stay limited to:

- status labels
- safe paths
- row counts
- case counts
- diagnostic count totals
- `content_suppressed=true`
- non-performance status fields

## 8. No-Oracle And Privacy Policy

The diagnostic distribution check is a diagnostic count validation.

It is not:

- a performance metric
- a tuning signal
- expected-action feedback
- a real-data validation
- a grammatical correctness check
- a learner-state estimate

The check must remain synthetic-only. It must not use real participant data,
post-edit text, final text, gold labels, teacher corrections, or expected
actions as scoring feedback.

## 9. Beginner Explanation

### What Is A Precondition?

A precondition is something that must be true before a command can work
correctly.

Here, the distribution check needs the summary file to already exist and contain
case rows.

### Why Must The Summary Run First?

The distribution check does not create the summary. It reads the summary.

If the summary does not exist yet, there are no cases to check.

### Why Can Parallel Execution Produce `no_cases`?

If one command is writing the summary while another command reads it, the reader
may see the file before it is complete.

That can look like an empty summary even though the generator would finish
successfully moments later.

### Why Not Treat `no_cases` As A Silent Pass?

Because `no_cases` can also mean something is genuinely wrong, such as summary
generation failing or producing no rows.

It should be interpreted using ordering context, not ignored.
