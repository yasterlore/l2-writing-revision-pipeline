# Forbidden-Term Path-Safety Test Hardening Design

This document designs how forbidden-term checks should distinguish content from
environment-dependent filesystem paths, and records the minimal test-helper
implementation added after that design.

It does not change scorer logic, scorer weights, scoring formula, deterministic
tie-break behavior, E2E pipeline logic, or any fixture.

This is not performance evaluation.

## 1. Purpose

The purpose of this document is to clarify the boundary for forbidden-term
checks.

The design should:

- keep no-oracle and privacy checks strict
- keep output-safety checks strict
- avoid false positives caused by environment-dependent temporary paths
- avoid treating full absolute paths as content bodies
- preserve rejection of real raw body leaks
- avoid adding F1, accuracy, calibration, or learner-state estimation

Step 105 implemented the first test-only helper for this design.

## 1.1 Implementation Status

The current implementation is intentionally narrow:

- `python/test_support/safe_output_scan.py` provides
  `normalize_environment_paths_for_scan(...)`
- stdout/stderr-oriented safe-output assertions can normalize paths before
  scanning forbidden fragments
- generated JSONL/report bodies are not blanket-normalized
- tests cover a temporary path containing a forbidden-looking substring and a
  real forbidden body term that still fails
- project-controlled output basenames remain visible to the scan

This helper is for tests only. It does not alter production output, scorer
behavior, E2E behavior, or config behavior.

## 2. Problem Summary

Some tests scan command output for forbidden terms.

A transient failure can happen when an OS-generated temporary directory or
temporary filename happens to contain a forbidden-looking substring. This is an
environment-dependent path issue, not evidence that a generated report body,
documentation body, JSONL body, config body, or score row leaked forbidden
content.

The same command may pass on rerun because a different temporary path is
generated.

This flakiness does not mean forbidden-term checks should be removed. The checks
are important. The boundary should be tightened so paths and content bodies are
handled differently.

## 3. Content That Should Be Checked

Forbidden-term scans should continue to check:

- committed documentation body
- generated report body
- stdout generated by project scripts, after path normalization when needed
- stderr generated by project scripts, after path normalization when needed
- generated summary field names
- config validation summaries
- candidate score output fields when explicitly intended
- committed files
- safe report text
- public release checklist text

If raw JSONL bodies, config bodies, summary bodies, diagnostic summary bodies,
candidate score rows, or private data appear in docs or unsafe output, the
check should still fail.

## 4. Content To Exclude Or Normalize

Forbidden-term scans should exclude or normalize:

- temporary directory absolute paths
- OS-generated temp names
- CI workspace root path
- user home path
- Python unittest temp file names
- tool-generated ephemeral path segments
- sandbox-specific temp paths
- machine-specific absolute prefixes

Caution: if a path basename is a project-controlled output name, review it
separately. Project-controlled names can communicate policy and should not be
automatically ignored.

## 5. Path Normalization Policy

Before scanning stdout or stderr, normalize known path-like substrings:

- absolute temp paths to `<TMP_PATH>`
- user home path to `<HOME>`
- CI workspace root to `<WORKSPACE>`
- project root to `<PROJECT_ROOT>`
- sandbox temp roots to `<TMP_PATH>`

After path normalization, run the forbidden-term scan on stdout and stderr.

Raw generated file contents should not be blanket-normalized. If the file body
is being checked, inspect the actual file content according to the relevant
safe-output policy.

Path normalization is for command output and ephemeral environment strings, not
for hiding body leaks.

## 6. Forbidden-Term Scan Priority

Recommended priority:

1. Raw committed docs content.
2. Generated report body.
3. Stdout after path normalization.
4. Stderr after path normalization.
5. File names reviewed separately.
6. Full absolute paths treated as environment metadata, not content bodies.

This order keeps real content checks strict while reducing path flakiness.

## 7. Important Exceptions And Context

Some terms can appear safely in policy or non-goal contexts.

Examples:

- documentation filenames may contain phrases such as
  `private_real_data_readiness_checklist.md` in a safety-policy context
- F1, accuracy, and calibration may appear as non-goals
- forbidden terms may appear in checklists that explicitly say not to implement
  or claim them

The scanner should distinguish between:

- unsafe body leaks
- policy/non-goal mentions
- generated field names
- path segments
- documentation filenames

Raw JSONL bodies, candidate score rows, config JSON bodies, summary bodies, and
diagnostic summary bodies in docs should still be rejected.

## 8. Future Test Hardening Direction

Future hardening may include:

- additional helper coverage for path normalization edge cases
- additional tests for temp paths containing forbidden-looking substrings
- additional tests proving real forbidden body content still fails
- tests proving policy/non-goal mentions are allowed only in approved contexts
- tests for project-controlled basenames
- avoiding broad regex scans on unnormalized absolute paths
- keeping output safety strict
- documenting the normalization boundary in test helper comments

The implementation should be small and focused. It should not weaken the
forbidden-term policy.

For a focused audit of where the current helper should and should not be
adopted, see
[safe output scan helper adoption audit](safe_output_scan_helper_adoption_audit.md).

## 9. What Not To Do Yet

Do not:

- disable forbidden-term checks
- broadly allow forbidden terms
- allow raw body leaks
- use path normalization to hide content leaks
- add F1
- add accuracy
- add calibration
- add learner-state estimation
- add performance claims
- change scorer weights
- change the scoring formula
- change tie-break behavior
- change config fixtures

## 10. Beginner Explanation

### What Is A False Positive?

A false positive is a failure that looks like a real problem but is caused by
something harmless.

Here, a temporary path can accidentally contain text that looks like a forbidden
term.

### What Is The Difference Between A Path And File Content?

A path is where a file lives. File content is what the file says.

Forbidden-term checks usually care about what the project prints or stores as
content, not random characters in OS-generated temporary directory names.

### Why Normalize Paths Instead Of Weakening Checks?

Normalizing paths removes environment noise while keeping real body checks
strict.

This helps avoid flaky tests without allowing private or no-oracle-unsafe
content to pass.

### Why Are Temporary Paths Environment-Dependent?

Operating systems and test frameworks generate temporary paths automatically.
Those names can vary on every run.

### Why Can A Rerun Pass?

On rerun, the temporary directory name may be different. If the new name does
not contain the forbidden-looking substring, the same test can pass.
